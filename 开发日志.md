## 2026-07-19（第一天：阶段 0 Spike）

### 决策记录

**技术栈反复了一次：Flutter → SwiftUI → Flutter（最终）**
- 最初定 Flutter（一套代码出 iOS + Android）。
- 中途因"只有 Mac + iPhone，追求最快跑通"短暂切到 SwiftUI，Flutter 安装都停了。
- 用户最终拍板回 Flutter，保留安卓可能性。
- **教训**：技术栈这种地基决策，应该在动手前用"将来要不要安卓"这一个问题定死，反复一次的代价是重写文档、重排任务、废掉一次安装。好在引擎层（C/C++ + C 接口封装）与 UI 栈无关，这部分工作完全没浪费——**把与平台无关的核心逻辑放在 C 接口后面，是天然的防反复保险**。

**引擎选型：Rapfi（GPLv3）**
- 五子棋最强开源引擎之一，CMake + C++17，自带 NEON（arm64）支持，正好覆盖 Mac/iPhone。
- 意外收获：引擎输出自带 `MESSAGE Eval xxx` 评分和 `MESSAGE Bestline ...` 主变，**阶段 2 的复盘功能核心数据引擎白送了**，不用额外开发评估接口。
- 默认构建用内置经典评估（不依赖外部神经网络权重文件），spike 阶段免去权重文件分发问题。

### 踩坑记录

**坑 1：xcode-select 指向 CommandLineTools，切换需要 sudo**
- 现象：装了完整 Xcode，但 `xcodebuild` 报错 "requires Xcode, but active developer directory is a command line tools instance"。
- 常规解法 `sudo xcode-select -s ...` 需要密码，自动化流程里跑不了。
- **更好的解法：`export DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer`**，环境变量级别覆盖，不需要 sudo，每次构建命令带上即可。

**坑 2：Xcode 26 的 iOS 平台组件是单独下载的（约 8GB）**
- SDK 本身随 Xcode 装好了（能编译），但模拟器运行时和真机构建目标（destination）都要 `xcodebuild -downloadPlatform iOS` 单独下载。
- 不下载的话：模拟器列表为空；`flutter build ios` 报 "iOS 26.0 is not installed"——**连真机构建都过不去，不只是模拟器的事**。
- 经验：环境搭建第一步就把这个下载挂到后台，它是整条链路上最长的等待项。

**坑 3：Rapfi 源码复用时的依赖取舍**
- Rapfi 的 CMake 只定义了可执行文件目标，没有库目标。做法：自己写 CMakeLists，取其 `CORE_SOURCES` 列表（去掉 `main.cpp`），编成静态库 `rapfi_core`。
- `command/argutils.cpp` 依赖 `cxxopts`（头文件库）——虽然它属于 CLI 参数解析，但被核心源码引用，直接把 `external/cxxopts` 加进依赖即可，不用改引擎源码。
- **原则：不改上游源码**（submodule 保持干净），所有适配放在自己的 wrapper 目录里，将来 `git submodule update` 升级引擎无痛。

**坑 4：iOS 交叉编译时 NEON 宏没生效，静态断言炸了**
- 现象：`static assertion failed: Failed to find a supported instruction set`（引擎的 NNUE 代码不允许纯标量编译）。
- 原因：交叉编译 iOS 时 `CMAKE_SYSTEM_PROCESSOR` 不是 "arm64"，我们的 `USE_NEON` 条件没匹配上。
- 解法：判断条件加上 `CMAKE_OSX_ARCHITECTURES MATCHES "arm64"` 和 `CMAKE_SYSTEM_NAME STREQUAL "iOS"`。
- **通用经验：凡是"宿主检测"逻辑（SIMD、字长、平台特性），交叉编译时都要重新审视**，宿主机检测结果对目标平台无效。

**坑 5：引擎 stdin/stdout 的进程内接管方式**
- 引擎是命令行程序思路：`gomocupLoop()` 读 std::cin、写 std::cout。iOS 不允许 fork/exec 子进程，必须进程内跑。
- 直觉方案 `dup2()` 管道会**劫持整个进程的 stdout**——宿主 App 自己的 print/日志全被吞掉，调试时是灾难。
- 采用的方案：**只换 C++ 流的 rdbuf**（`std::cin.rdbuf(...)` / `std::cout.rdbuf(...)`），换成自写的线程安全内存管道 streambuf。C 层 stdio 不动，Swift/Dart 的打印完全不受影响。
- 引擎协议循环跑在 `std::thread` 上，对外暴露三个 C 函数：`rapfi_start()` / `rapfi_send(line)` / `rapfi_recv(buf, cap, timeout_ms)`。

**坑 6：C++ 编译缺 `<iostream>`**
- `std::cin/cout` 报 "no member named 'cin' in namespace 'std'"，低级但常见：`<streambuf>` 等头文件不会连带引入 `<iostream>`。

### FFI / Flutter 集成要点（写在前面，0.6 验证时对照）

- 静态库通过 xcconfig 链入 Runner：`OTHER_LDFLAGS` 带 **`-force_load`**（否则链接器会把"没被 Swift 代码引用"的桥接符号裁掉，Dart FFI 运行时 `lookup` 找不到符号）+ `-lc++`。
- 真机和模拟器是**两个不同 SDK 的静态库**，用 `[sdk=iphoneos*]` / `[sdk=iphonesimulator*]` 条件分别指向 `build-ios/` 和 `build-ios-sim/`。
- Dart 侧 `rapfi_recv` 是阻塞调用，**必须放 `Isolate.run()` 里**，否则引擎思考 2 秒 UI 冻 2 秒。
- 待验证风险：Release 构建的符号裁剪（strip）可能影响 `DynamicLibrary.process()` 的符号查找，Debug 没问题；真机 Release 如果出问题，解法是 Strip Style 改 Non-Global Symbols。

**坑 7：静态库不会带上自己的依赖——模拟器链接报 lz4 符号缺失**
- 现象：`flutter build ios --no-codesign`（真机 Release）通过，但模拟器构建报 `Undefined symbol: _LZ4F_*`。
- 原因：`librapfi_core.a` 对 `liblz4.a` 是 PRIVATE 链接，**静态库只是 .o 的归档，不会嵌入它依赖的其他静态库**。
- 解法：xcconfig 的 `OTHER_LDFLAGS` 里把 `liblz4.a` 一起列上。
- 经验：静态库集成时，把 CMake 里所有非 header-only 的依赖库（这里只有 lz4；cpptoml/cxxopts/simde 均为纯头文件）逐个找出来一起链；报错缺什么符号就 `find build-ios -name "*.a"` 找归属。

**0.6 验收方式：集成测试代替人肉点击**
- 用 `integration_test` 写了一条用例：真实启动 App → `tester.tap` 按钮 → 轮询断言出现 `引擎开局: \d+,\d+`。
- 好处：验收可重复、可进 CI，比截图硬得多。模拟器上 1 秒内通过。

**坑 8：Release 构建死代码消除删掉了 FFI 符号（真机首跑报 symbol not found）**
- 现象：Debug/模拟器一切正常；真机 Release 版点按钮报 `Failed to lookup symbol 'rapfi_start': dlsym(RTLD_DEFAULT, ...): symbol not found`。
- 原因链：FFI 符号是运行时 `dlsym` 查找的，编译期没有任何 Swift/Dart 引用 → Release 开启 dead-code stripping，链接器认为三个桥接函数"没人用"直接删除（`-force_load` 只保证归档里所有 .o 参与链接，管不住后续的死代码消除）。
- 解法（两层保险）：
  1. 函数声明加 `__attribute__((used)) __attribute__((visibility("default")))`（头文件里定义成 `RAPFI_EXPORT` 宏）——阻止死代码消除并保证进动态符号表；
  2. xcconfig 加 `STRIP_STYLE = non-global`——阻止 strip 阶段剥掉全局符号。
- 验证手段：`nm -gU Runner.app/Runner | grep rapfi`，三个符号都要是 `T`（全局导出）。
- **通用经验：凡是"只被运行时查找"的符号（FFI、dlopen 插件、反射），必须显式标记导出**，Debug 正常不代表 Release 正常。

**真机部署流程备忘（免费 Apple ID）**
1. iPhone 连线 + 信任电脑；设置 → 隐私与安全性 → 开发者模式（要重启）。
2. Xcode 登录 Apple ID 后，**必须在 Runner 的 Signing & Capabilities 里选一次 Team**，第一张开发证书才会创建（纯命令行绕不过）。
3. 手机上信任开发者证书：设置 → 通用 → VPN 与设备管理。
4. Debug 包从主屏幕点图标会闪退（Dart JIT 需要调试器），**独立使用必须装 Release 包**。
5. Flutter 偶发"Build succeeded but expected app not found"：产物实际在 `build/ios/Release-iphoneos/`，直接 `xcrun devicectl device install app` + `devicectl device process launch` 即可，不用纠结 flutter run。
6. 免费签名 7 天过期，过期后重新装一次即可。

### 工程习惯

- 每个 step 先明确验收标准再动手（见 [实施步骤.md](./实施步骤.md)），做完立即验证，不攒着。
- 大下载（Flutter SDK、iOS 平台组件、CocoaPods）全部后台并行，前台继续推进不依赖它们的 step——今天靠这个把等待时间几乎全部利用起来了。
- 上游引擎用 git submodule，改动一律放自己目录，保持升级通道干净。
