## 2026-07-19（第一天：阶段 0 Spike）

### 决策记录

**技术栈反复了一次：Flutter → SwiftUI → Flutter（最终）**
- 最初定 Flutter（一套代码出 iOS + Android）。
- 中途因"只有 Mac + iPhone，追求最快跑通"短暂切到 SwiftUI，Flutter 安装都停了。
- 用户最终拍板回 Flutter，保留安卓可能性。
- **教训**：技术栈这种地基决策，应该在动手前用"将来要不要安卓"这一个问题定死，反复一次的代价是重写文档、重排任务、废掉一次安装。好在引擎层（C/C++ + C 接口封装）与 UI 栈无关，这部分工作完全没浪费——**把与平台无关的核心逻辑放在 C 接口后面，是天然的防反复保险**。

**引擎选型：Rapfi（GPLv3）**
- 五子棋最强开源引擎之一，CMake + C++17，自带 NEON（arm64）支持，正好覆盖 Mac/iPhone。
- 意外收获：引擎输出自带 `MESSAGE Eval xxx` 评分和 `MESSAGE Bestline ...` 主变，**阶段 2 的复盘功能核心数据引擎白送了**，不用额外开发评估接口。
- 默认构建用内置经典评估（不依赖外部神经网络权重文件），spike 阶段免去权重文件分发问题。

### 踩坑记录

**坑 1：xcode-select 指向 CommandLineTools，切换需要 sudo**
- 现象：装了完整 Xcode，但 `xcodebuild` 报错 "requires Xcode, but active developer directory is a command line tools instance"。
- 常规解法 `sudo xcode-select -s ...` 需要密码，自动化流程里跑不了。
- **更好的解法：`export DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer`**，环境变量级别覆盖，不需要 sudo，每次构建命令带上即可。

**坑 2：Xcode 26 的 iOS 平台组件是单独下载的（约 8GB）**
- SDK 本身随 Xcode 装好了（能编译），但模拟器运行时和真机构建目标（destination）都要 `xcodebuild -downloadPlatform iOS` 单独下载。
- 不下载的话：模拟器列表为空；`flutter build ios` 报 "iOS 26.0 is not installed"——**连真机构建都过不去，不只是模拟器的事**。
- 经验：环境搭建第一步就把这个下载挂到后台，它是整条链路上最长的等待项。

**坑 3：Rapfi 源码复用时的依赖取舍**
- Rapfi 的 CMake 只定义了可执行文件目标，没有库目标。做法：自己写 CMakeLists，取其 `CORE_SOURCES` 列表（去掉 `main.cpp`），编成静态库 `rapfi_core`。
- `command/argutils.cpp` 依赖 `cxxopts`（头文件库）——虽然它属于 CLI 参数解析，但被核心源码引用，直接把 `external/cxxopts` 加进依赖即可，不用改引擎源码。
- **原则：不改上游源码**（submodule 保持干净），所有适配放在自己的 wrapper 目录里，将来 `git submodule update` 升级引擎无痛。

**坑 4：iOS 交叉编译时 NEON 宏没生效，静态断言炸了**
- 现象：`static assertion failed: Failed to find a supported instruction set`（引擎的 NNUE 代码不允许纯标量编译）。
- 原因：交叉编译 iOS 时 `CMAKE_SYSTEM_PROCESSOR` 不是 "arm64"，我们的 `USE_NEON` 条件没匹配上。
- 解法：判断条件加上 `CMAKE_OSX_ARCHITECTURES MATCHES "arm64"` 和 `CMAKE_SYSTEM_NAME STREQUAL "iOS"`。
- **通用经验：凡是"宿主检测"逻辑（SIMD、字长、平台特性），交叉编译时都要重新审视**，宿主机检测结果对目标平台无效。

**坑 5：引擎 stdin/stdout 的进程内接管方式**
- 引擎是命令行程序思路：`gomocupLoop()` 读 std::cin、写 std::cout。iOS 不允许 fork/exec 子进程，必须进程内跑。
- 直觉方案 `dup2()` 管道会**劫持整个进程的 stdout**——宿主 App 自己的 print/日志全被吞掉，调试时是灾难。
- 采用的方案：**只换 C++ 流的 rdbuf**（`std::cin.rdbuf(...)` / `std::cout.rdbuf(...)`），换成自写的线程安全内存管道 streambuf。C 层 stdio 不动，Swift/Dart 的打印完全不受影响。
- 引擎协议循环跑在 `std::thread` 上，对外暴露三个 C 函数：`rapfi_start()` / `rapfi_send(line)` / `rapfi_recv(buf, cap, timeout_ms)`。

**坑 6：C++ 编译缺 `<iostream>`**
- `std::cin/cout` 报 "no member named 'cin' in namespace 'std'"，低级但常见：`<streambuf>` 等头文件不会连带引入 `<iostream>`。

### FFI / Flutter 集成要点（写在前面，0.6 验证时对照）

- 静态库通过 xcconfig 链入 Runner：`OTHER_LDFLAGS` 带 **`-force_load`**（否则链接器会把"没被 Swift 代码引用"的桥接符号裁掉，Dart FFI 运行时 `lookup` 找不到符号）+ `-lc++`。
- 真机和模拟器是**两个不同 SDK 的静态库**，用 `[sdk=iphoneos*]` / `[sdk=iphonesimulator*]` 条件分别指向 `build-ios/` 和 `build-ios-sim/`。
- Dart 侧 `rapfi_recv` 是阻塞调用，**必须放 `Isolate.run()` 里**，否则引擎思考 2 秒 UI 冻 2 秒。
- 待验证风险：Release 构建的符号裁剪（strip）可能影响 `DynamicLibrary.process()` 的符号查找，Debug 没问题；真机 Release 如果出问题，解法是 Strip Style 改 Non-Global Symbols。

**坑 7：静态库不会带上自己的依赖——模拟器链接报 lz4 符号缺失**
- 现象：`flutter build ios --no-codesign`（真机 Release）通过，但模拟器构建报 `Undefined symbol: _LZ4F_*`。
- 原因：`librapfi_core.a` 对 `liblz4.a` 是 PRIVATE 链接，**静态库只是 .o 的归档，不会嵌入它依赖的其他静态库**。
- 解法：xcconfig 的 `OTHER_LDFLAGS` 里把 `liblz4.a` 一起列上。
- 经验：静态库集成时，把 CMake 里所有非 header-only 的依赖库（这里只有 lz4；cpptoml/cxxopts/simde 均为纯头文件）逐个找出来一起链；报错缺什么符号就 `find build-ios -name "*.a"` 找归属。

**0.6 验收方式：集成测试代替人肉点击**
- 用 `integration_test` 写了一条用例：真实启动 App → `tester.tap` 按钮 → 轮询断言出现 `引擎开局: \d+,\d+`。
- 好处：验收可重复、可进 CI，比截图硬得多。模拟器上 1 秒内通过。

**坑 8：Release 构建死代码消除删掉了 FFI 符号（真机首跑报 symbol not found）**
- 现象：Debug/模拟器一切正常；真机 Release 版点按钮报 `Failed to lookup symbol 'rapfi_start': dlsym(RTLD_DEFAULT, ...): symbol not found`。
- 原因链：FFI 符号是运行时 `dlsym` 查找的，编译期没有任何 Swift/Dart 引用 → Release 开启 dead-code stripping，链接器认为三个桥接函数"没人用"直接删除（`-force_load` 只保证归档里所有 .o 参与链接，管不住后续的死代码消除）。
- 解法（两层保险）：
  1. 函数声明加 `__attribute__((used)) __attribute__((visibility("default")))`（头文件里定义成 `RAPFI_EXPORT` 宏）——阻止死代码消除并保证进动态符号表；
  2. xcconfig 加 `STRIP_STYLE = non-global`——阻止 strip 阶段剥掉全局符号。
- 验证手段：`nm -gU Runner.app/Runner | grep rapfi`，三个符号都要是 `T`（全局导出）。
- **通用经验：凡是"只被运行时查找"的符号（FFI、dlopen 插件、反射），必须显式标记导出**，Debug 正常不代表 Release 正常。

**真机部署流程备忘（免费 Apple ID）**
1. iPhone 连线 + 信任电脑；设置 → 隐私与安全性 → 开发者模式（要重启）。
2. Xcode 登录 Apple ID 后，**必须在 Runner 的 Signing & Capabilities 里选一次 Team**，第一张开发证书才会创建（纯命令行绕不过）。
3. 手机上信任开发者证书：设置 → 通用 → VPN 与设备管理。
4. Debug 包从主屏幕点图标会闪退（Dart JIT 需要调试器），**独立使用必须装 Release 包**。
5. Flutter 偶发"Build succeeded but expected app not found"：产物实际在 `build/ios/Release-iphoneos/`，直接 `xcrun devicectl device install app` + `devicectl device process launch` 即可，不用纠结 flutter run。
6. 免费签名 7 天过期，过期后重新装一次即可。

## 2026-07-19（第二天：阶段 1 五子棋 MVP）

### 踩坑记录

**坑 9：`path_provider` 新版拉入 Dart Native Assets 钩子，在 Xcode Run Script 阶段里 `xcrun` 找不到 SDK**
- 现象：加了 `path_provider`（为 1.6 对局记录做文件落盘）后，`flutter test integration_test/...` 在模拟器上构建报 `Bad state: No element`，源头是 `objective_c` 包（`path_provider_foundation` ≥ 2.5 的传递依赖）的 native-assets build hook 里 `Process.runSync('xcrun', ['--show-sdk-path', '--sdk', 'iphonesimulator'])` 拿到空 stdout。
- 排查：命令行里手动 `DEVELOPER_DIR=... xcrun --show-sdk-path --sdk iphonesimulator` 是正常的；说明不是我们外层 shell 的 `DEVELOPER_DIR` 没设对（坑 1 的老问题），而是 **Xcode 的 Run Script 构建阶段会用自己的环境跑这个 native-assets 钩子**，而当时 `xcode-select -p` 全局指向的还是 CommandLineTools，这个钩子进程没有拿到我们外层导出的 `DEVELOPER_DIR`。
- 解法（绕过，未解决根因）：`pubspec.yaml` 里 `dependency_overrides: path_provider_foundation: 2.4.1`，钉回还在用纯 Cocoapods（不用 native-assets 钩子）的版本，问题消失。
- **通用经验**：Dart/Flutter 的 Native Assets（`hook/build.dart`）是新机制，构建时机在 Xcode 的 Run Script 阶段而非我们主动调用的 `flutter build` 进程本身，**外层 shell 导出的环境变量不一定能传导进去**；如果某个包的 native-assets 钩子在 CI/沙盒环境里报环境相关的错，先看是不是能锁一个不用该机制的旧版本绕过，比死磕 Xcode 环境变量传递链路更省事。

**坑 10（非 bug，设计确认）：`SizedBox` 在紧约束的父级下会忽略自己声明的宽高**
- 现象：`BoardView` 组件测试直接把 `SizedBox(width: 300, height: 300, child: BoardView(...))` 塞进 `MaterialApp.home`，结果点击坐标全部算错。
- 原因：`MaterialApp.home` 拿到的是路由撑满整个测试 surface 的**紧约束**（比如测试默认 800×600），`SizedBox` 在紧约束下无法覆盖父级尺寸，实际渲染成了 800×600 而不是 300×300。
- 解法：测试里给 `SizedBox` 套一层 `Center`（给它一个宽松约束），并且用 `tester.getRect` 读实际渲染尺寸算点击坐标，不要硬编码假设的边长。`game_page.dart` 里本来就是 `Expanded(child: Center(child: BoardView(...)))`，这个写法本身没问题，纯粹是补测试时漏了 `Center`。

## 2026-07-19（第三天：阶段 2 复盘功能）

### 踩坑记录

**坑 11：15×15 空棋盘走开局库，没有 Eval 输出**
- 现象：`GameReviewer` 用 `BOARD` 命令查询空棋盘（第 0 手之前）的评分时，日志里只有落子坐标（如 `7,7`），没有期待的 `MESSAGE Speed ... Eval ...` 行，导致该局面评分解析失败。
- 排查：直接读 `search/opening.cpp`，`generateOpening()` 对 FREEOPEN 规则的 `board.ply() == 0` 情形直接返回 `board.centerPos()`（固定天元），完全跳过搜索——这是引擎的既定行为，不是 bug，且协议里没有开关能关掉它（`disableOpeningQuery` 只是内部 C++ 参数，没有对应的 Gomocup 文本命令）。
- 解法：`GameReviewer._evalPosition` 里把空棋盘（`i == 0`）直接当作中性局面（`score: 0`），不查引擎，跟"终局已经五连/已经下满"两个特殊局面一样处理。
- **通用经验**：给协议型引擎做"查任意局面评分"这种非常规用法时，先假设它对边界局面（空局面、终局）有特殊快捷路径，不要默认每次查询都会触发完整搜索——最好先写一个最小化的探测脚本（或者像这次一样直接读引擎源码里 `ply() == 0` 这类分支）确认实际行为，而不是等测试报"评分缺失"才去猜。

**协议细节：`Bestline` 用棋盘记号（`H8`），不是 `x,y`**
- `MESSAGE Bestline` 输出的走法列表是 `operator<<(ostream&, Pos)` 格式（列字母 + 1-indexed 行号，比如 `H8`），跟落子应答用的 `x,y` 格式（`CoordText`）是两套不同的坐标序列化。复盘用到 PV 的地方（`engine_value.dart` 的 `parseLabelCoord`）专门写了一个单独的解析函数，不能和处理落子应答的正则复用。

## 2026-07-19（第四天：阶段 4.1 引入 Pikafish 象棋引擎）

> 用户明确要求"以后不用问我，自己推，剩下的全部给我做了"——阶段 4 起不再逐个 step 确认，直接做完整段再汇报。

### 决策记录

**引擎选型细化：Pikafish 实际是 UCI，不是 UCCI**
- 技术方案.md 原先写"象棋走 UCCI 协议"，是笼统参考中国象棋引擎圈的常见说法。实测 Pikafish（Stockfish 分支）README/协议文档直接说自己是标准 **UCI** 引擎，命令就是 `uci`/`isready`/`position`/`go`/`bestmove` 这套，跟五子棋引擎接的协议不是一回事，但**跟国际象棋引擎是一回事**——不需要额外做 UCCI 转译层，直接照搬熟悉的 UCI 客户端写法即可。

### 踩坑记录

**坑 12：绕过 Makefile 直接编译 Pikafish，`Bitboard` 类型报"未声明"**
- 现象：`native/pikafish/src/bitboard.h` 里 `constexpr Bitboard Palace = ...` 报 `unknown type name 'Bitboard'`。
- 排查：`Bitboard` 在 `types.h` 里是 `using Bitboard = u128;`，而 `u128` 定义在 `misc.h` 里，套着 `#if defined(__GNUC__) && defined(IS_64BIT)` 条件编译——`IS_64BIT` 这个宏是 Pikafish 自己 Makefile 在 `bits=64` 时通过 `-DIS_64BIT` 传进来的，我们绕开 Makefile 直接用 CMake 编译，这个宏没人定义，`u128`/`Bitboard` 整个不存在。
- 解法：对照 Makefile 里 `ARCH=apple-silicon` 分支实际展开的宏（`bits=64 → -DIS_64BIT`；`popcnt=yes` 且 arch 在 arm 系列白名单里 → `-DUSE_POPCNT`；`neon=yes` → `-DUSE_NEON=$(arm_version)`，apple-silicon 的 `arm_version=8`），在 CMakeLists 里手动 `target_compile_definitions(pikafish_core PRIVATE IS_64BIT USE_POPCNT USE_NEON=8)` 补上。`dotprod=yes` 对应的 `-march=armv8.2-a+dotprod` 先没加（iOS 工具链下 `-march` 行为没验证过，先跑通功能，性能调优后置）。
- **通用经验**：跟 Rapfi 的坑 4（NEON 宏）是同一类问题的再次出现——**凡是绕开上游 Makefile/构建脚本、自己用 CMake 重新收编源码的场景，一定要通读一遍上游 Makefile 里"当前目标架构会展开成哪些宏定义"，不能只看头文件里出现的宏名字**，因为这些宏假设了"你用的是我的构建系统"，直接换构建系统会导致条件编译分支整体消失而不是报错走 fallback（`Bitboard` 类型直接找不到，而不是退化成 64 位版本）。

**坑 13：`-fno-exceptions` 是上游明确要求的编译选项，不是可选项**
- Pikafish 的 Makefile 固定带 `-fno-exceptions`（`CXXFLAGS = ... -fno-exceptions -std=c++17 ...`）。既然上游全部源码（含 vendor 进来的 zstd 解压库）都是在这个假设下写的，我们自己的 CMake 目标也照抄这个 flag，没有另外验证"不加会怎样"——直接对齐上游已验证过的构建配置，比自己去猜哪些文件用了 `throw` 更可靠。

**坑 14：NNUE 权重文件（~50MB）不进 git，运行时才拼装**
- Pikafish 需要一个 `pikafish.nnue` 权重文件才能跑 NNUE 评估（纯规则引擎跑不动，现代 Stockfish 系都是 NNUE-only，没有经典评估兜底）。文件本身不在源码仓库里，官方是运行时/构建时从 GitHub Release 下载（Pikafish 自己的 `scripts/net.sh` 就是这么干的）。
- 处理方式跟 `build-ios/` 静态库产物一路的思路：**不提交进 git**（50MB 二进制 + GitHub 单文件 100MB 硬限但 50MB 就会有软提示，没必要冒这个险），而是：
  1. `native/scripts/fetch-pikafish-net.sh` 一次性下载到 `app/assets/pikafish.nnue`（新建 Flutter assets 目录）；
  2. `pubspec.yaml` 把它声明成资源文件，随 App 一起打包（个人测试装机不受 App Store 体积限制，50MB 完全没问题）；
  3. 运行时 `PikafishAssets.ensureNetworkFile()` 把打包进 App 里的资源文件（在归档里，不是普通文件）解压一份到文档目录，再把这个可写路径通过 `setoption name EvalFile value <path>` 告诉引擎——因为 Pikafish 用 `fopen()` 直接读文件，读不了 Flutter 资源归档里的内容。
- **通用经验**：只要是"引擎运行时依赖的大体积外部数据文件"（不是源码、不是构建产物），处理思路统一是"写个一次性下载脚本 + gitignore + 运行时按需落地到可写目录"，跟阶段 0 处理 iOS 平台组件下载、这次处理静态库构建产物是同一套思路的自然延伸。
