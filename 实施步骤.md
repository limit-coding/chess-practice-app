# 实施步骤（分阶段 / 分节点）

> 承接 [技术方案.md](./技术方案.md)。本文档把项目拆成可以逐个交给模型执行的具体 step，每个 step 都写明**产出物**和**验收标准**，方便你在换模型（比如试用 Fable 5）时，按 step 对比不同模型做出来的东西质量如何。

先做**五子棋**这条线跑通全流程（引擎对弈 → 复盘 → 真机可用），再扩展到象棋。

## 阶段 0：环境与可行性验证（Spike）

> 2026-07-19 更新（二次）：技术栈最终定为 **Flutter**（用户拍板，保留安卓可能性；Swift/Xcode 仅作为 iOS 构建与真机安装的底层通道）。
>
> 执行进展：0.3、0.4 已完成——Rapfi 引擎在 Mac 上编译通过并正确走棋；C 桥接层（`native/wrapper/`，线程内跑引擎协议循环 + 内存管道，暴露 `rapfi_start/rapfi_send/rapfi_recv` 三个 C 函数）已写完，纯 C 测试程序全部 PASS。

目标：证明 Flutter 能通过 `dart:ffi` 调用 C/C++ 五子棋引擎拿到落子结果，并且能装到真机上运行。这是全项目风险最高的一步。

- **0.1 环境检查** ✅
  产出：Flutter + CocoaPods + Xcode 26（经 `DEVELOPER_DIR` 环境变量指定，无需 sudo）+ iOS 26 平台组件全部就绪。

- **0.2 创建项目骨架** ✅
  产出：`app/`（`flutter create`，包名 `dev.limit.chess_practice`，iOS + Android 平台）。

- **0.3 选定并引入五子棋引擎源码** ✅
  产出：Rapfi（GPLv3）作为 git submodule 放在 `native/rapfi/`。
  验收：已通过——Mac 原生编译成功，Gomocup 协议实测正常走棋（附带 Eval 评分和 Bestline 主变输出，复盘功能可直接复用）。

- **0.4 编写引擎的 C 接口封装（extern "C" wrapper）** ✅
  产出：`native/wrapper/rapfi_bridge.{h,cpp}`——引擎协议循环跑在进程内线程上，stdin/stdout 换成线程安全内存管道（不动 C 层 stdio，宿主的 print 不受影响），对外仅 `rapfi_start()` / `rapfi_send(line)` / `rapfi_recv(buf, cap, timeout_ms)` 三个 C 函数。
  验收：已通过——`native/wrapper/test_main.c`（纯 C）驱动引擎完成 START/BEGIN/TURN 全流程，ALL PASS。

- **0.5 引擎编译为 iOS 静态库** ✅
  产出：`build-ios/`（真机 arm64）与 `build-ios-sim/`（模拟器）两份 `librapfi_core.a`。

- **0.6 Dart 端 FFI 绑定与最小验证 UI** ✅
  产出：`app/lib/engine/rapfi_ffi.dart`（FFI 绑定，阻塞 recv 放独立 isolate）、`app/lib/main.dart`（验证 UI）、`app/ios/Flutter/Rapfi.xcconfig`（静态库链接，真机/模拟器分 SDK，附 lz4）。
  验收：已通过——`integration_test/engine_test.dart` 在模拟器上自动点击按钮，断言出现"引擎开局: x,y"，1 秒内 PASS。

- **0.7 真机验证** ✅
  产出：Release 版装上用户 iPhone（免费 Apple ID 签名，流程备忘见开发日志）。
  验收：已通过——真机点击按钮，引擎正常应对落子（还顺手报了个 +M3 必胜评估）。中途踩坑 Release 死代码消除删 FFI 符号，已修（开发日志坑 8）。

> **阶段 0 全部完成（2026-07-19）**：C++ 引擎 → C 桥接 → Dart FFI → Flutter UI → iPhone 真机，全链路打通。

> **阶段 0 完成标志**：你在自己的 iPhone 上，点一个按钮，能看到五子棋引擎算出的一步棋。

## 阶段 1：五子棋人机对弈 MVP

- **1.1 棋盘 UI**：15×15 网格棋盘，支持点击落子。验收：落子位置准确对应点击坐标。
- **1.2 游戏状态管理**：当前棋局、历史着法、当前该谁走。验收：状态在多次落子后保持一致，无错乱。
- **1.3 接入引擎对弈**：玩家落子后自动调用引擎计算 AI 应对并落子。验收：完整走完一整局不报错。
- **1.4 胜负判定**：五连检测，游戏结束提示。验收：横/竖/斜四个方向都能正确判定。
- **1.5 难度分级**：难度选项映射到引擎搜索深度/时间参数。验收：不同难度下引擎明显强弱有别。
- **1.6 对局记录**：每步棋 + 时间戳落盘，为复盘做数据准备。验收：一局下完后能读出完整的着法序列。

---

## 阶段 2：复盘功能

- **2.1 逐步引擎评估**：对局结束后，对每一步之前/之后的局面重新跑引擎评估。验收：能拿到一条完整的评分曲线。
- **2.2 着法分类**：按评分损失值分档（漏着/错误/不精确/好棋/最佳）。验收：明显的漏着（比如没挡住对方冲四）能被正确标记。
- **2.3 复盘界面**：评分曲线 + 逐步列表，标出问题着法。验收：点开复盘页能一眼看出这局哪几步最差。
- **2.4 残局回退**：点击某个问题着法，棋盘退回到该局面。验收：退回后的局面和实际对局完全一致。
- **2.5 正确下法提示**：展示引擎推荐着法（PV），用户可以照着走，并在此基础上继续对弈练习。验收：走完提示的一步后，能接着正常对弈下去。
- **2.6（可选，后置）自然语言解释**：在 2.1-2.5 都跑通之后，才考虑接入 LLM 把引擎结果转成人话解释。验收：解释内容不影响 2.1-2.5 的判断结果，纯展示层。

## 使用建议

- 每次只交给模型**一个 step**，做完检查验收标准是否达标，再继续下一个。
- 如果要对比模型（比如 Sonnet vs Fable 5），建议挑**同一个 step** 分别用不同模型各做一遍，在独立分支/目录下对比产出，而不是在同一条主线上来回切换模型——避免风格和实现方式互相污染，也更容易看出差异。
- 阶段 0 的每一步都建议自己盯着验收标准逐条确认，这是后面所有工作的地基。
