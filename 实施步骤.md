# 实施步骤（分阶段 / 分节点）

> 承接 [技术方案.md](./技术方案.md)。本文档把项目拆成可以逐个交给模型执行的具体 step，每个 step 都写明**产出物**和**验收标准**，方便你在换模型（比如试用 Fable 5）时，按 step 对比不同模型做出来的东西质量如何。

>
> 执行进展：0.3、0.4 已完成——Rapfi 引擎在 Mac 上编译通过并正确走棋；C 桥接层（`native/wrapper/`，线程内跑引擎协议循环 + 内存管道，暴露 `rapfi_start/rapfi_send/rapfi_recv` 三个 C 函数）已写完，纯 C 测试程序全部 PASS。

目标：证明 Flutter 能通过 `dart:ffi` 调用 C/C++ 五子棋引擎拿到落子结果，并且能装到真机上运行。这是全项目风险最高的一步。

- **0.1 环境检查** ✅
  产出：Flutter + CocoaPods + Xcode 26（经 `DEVELOPER_DIR` 环境变量指定，无需 sudo）+ iOS 26 平台组件全部就绪。

- **0.2 创建项目骨架** ✅
  产出：`app/`（`flutter create`，包名 `dev.limit.chess_practice`，iOS + Android 平台）。

- **0.3 选定并引入五子棋引擎源码** ✅
  产出：Rapfi（GPLv3）作为 git submodule 放在 `native/rapfi/`。
  验收：已通过——Mac 原生编译成功，Gomocup 协议实测正常走棋（附带 Eval 评分和 Bestline 主变输出，复盘功能可直接复用）。

- **0.4 编写引擎的 C 接口封装（extern "C" wrapper）** ✅
  产出：`native/wrapper/rapfi_bridge.{h,cpp}`——引擎协议循环跑在进程内线程上，stdin/stdout 换成线程安全内存管道（不动 C 层 stdio，宿主的 print 不受影响），对外仅 `rapfi_start()` / `rapfi_send(line)` / `rapfi_recv(buf, cap, timeout_ms)` 三个 C 函数。
  验收：已通过——`native/wrapper/test_main.c`（纯 C）驱动引擎完成 START/BEGIN/TURN 全流程，ALL PASS。

- **0.5 引擎编译为 iOS 静态库** ✅
  产出：`build-ios/`（真机 arm64）与 `build-ios-sim/`（模拟器）两份 `librapfi_core.a`。

- **0.6 Dart 端 FFI 绑定与最小验证 UI** ✅
  产出：`app/lib/engine/rapfi_ffi.dart`（FFI 绑定，阻塞 recv 放独立 isolate）、`app/lib/main.dart`（验证 UI）、`app/ios/Flutter/Rapfi.xcconfig`（静态库链接，真机/模拟器分 SDK，附 lz4）。
  验收：已通过——`integration_test/engine_test.dart` 在模拟器上自动点击按钮，断言出现"引擎开局: x,y"，1 秒内 PASS。

- **0.7 真机验证** ✅
  产出：Release 版装上用户 iPhone（免费 Apple ID 签名，流程备忘见开发日志）。
  验收：已通过——真机点击按钮，引擎正常应对落子（还顺手报了个 +M3 必胜评估）。中途踩坑 Release 死代码消除删 FFI 符号，已修（开发日志坑 8）。

> **阶段 0 全部完成（2026-07-19）**：C++ 引擎 → C 桥接 → Dart FFI → Flutter UI → iPhone 真机，全链路打通。

## 阶段 1：五子棋人机对弈 MVP

> 2026-07-19 完成。玩家固定执黑先手，引擎固定执白应对（对应 `native/wrapper` 的 TURN 协议）；棋盘/状态/胜负判定/难度/对局记录分别落在 `app/lib/widgets/board_view.dart`、`app/lib/game/game_state.dart`、`app/lib/game/win_checker.dart`、`app/lib/game/difficulty.dart`、`app/lib/game/game_record.dart`，由 `app/lib/pages/game_page.dart` 串起来。

- **1.1 棋盘 UI** ✅：15×15 网格棋盘，支持点击落子。验收：落子位置准确对应点击坐标——`test/board_view_test.dart` 覆盖 225 个交叉点的坐标往返映射 + 一次真实 tap 命中断言。
- **1.2 游戏状态管理** ✅：当前棋局、历史着法、当前该谁走。验收：状态在多次落子后保持一致，无错乱——`test/game_state_test.dart`。
- **1.3 接入引擎对弈** ✅：玩家落子后自动调用引擎计算 AI 应对并落子。验收：完整走完一整局不报错——`integration_test/engine_test.dart` 在模拟器上跑通一整回合（玩家落子 → FFI 拿到引擎应对 → 无 ERROR/超时），PASS。
- **1.4 胜负判定** ✅：五连检测，游戏结束提示。验收：横/竖/斜四个方向都能正确判定——`test/win_checker_test.dart` 逐方向覆盖，另加连四不算赢、对方棋子不接线、长连（六子）仍算赢的边界情形。
- **1.5 难度分级** ✅：难度选项（简单/中等/困难）映射到 `INFO TIMEOUT_TURN` + `INFO MAX_DEPTH`。验收：不同难度下引擎明显强弱有别——`test/difficulty_test.dart` 断言难度递增时思考时间与搜索深度单调不减；真实强弱差异需实测棋局体感，留待日常使用中检验。
- **1.6 对局记录** ✅：每步棋 + 时间戳以 JSON 落盘到 App 文档目录 `games/`。验收：一局下完后能读出完整的着法序列——`test/game_record_test.dart` 存档后重新读取，逐步比对着法与执子颜色。

踩坑记录见 开发日志.md（`path_provider` 引入的 native-assets 构建问题）。

## 阶段 2：复盘功能

> 2026-07-19 完成（2.1-2.5；2.6 按计划后置，未做）。核心是 Gomocup 协议里的 `BOARD` 命令——能用 `x,y,1/2`（1=先手/黑，2=后手/白）重建任意历史局面并触发引擎搜索，返回的 `MESSAGE ... Eval <val>`（支持 `+M<n>` 杀棋记号）和 `MESSAGE Bestline <H8 G8 ...>`（字母数字坐标，不是 x,y！）就是复盘的核心数据源。逻辑在 `app/lib/game/{engine_value,game_review}.dart`，界面在 `app/lib/pages/review_page.dart` + `app/lib/widgets/score_curve.dart`。

- **2.1 逐步引擎评估** ✅：对局结束后，对每一步之前的局面用 `BOARD` 命令重新跑引擎评估（复盘强度固定，不受当局难度影响）。验收：能拿到一条完整的评分曲线——`integration_test/review_test.dart` 在模拟器上用真实引擎跑通，7 个局面全部拿到评分，无缺失。踩坑：15×15 空棋盘引擎走开局库直接落子，不搜索、不输出 Eval，特殊处理成中性 0 分（详见开发日志）。
- **2.2 着法分类** ✅：negamax 损失公式 `loss = before.score + after.score`（同色视角前后局面评分之和），按损失分五档（最佳/好棋/不精确/错误/漏着）。验收：明显漏着能被正确标记——`test/game_review_test.dart` 覆盖阈值边界、噪声钳位为 0、送出必杀的巨大损失三类场景。注意阈值是 Rapfi 自己的评分量纲（±30000 为杀棋区间），不是 centipawn，是未经大量实战数据校准的初版，标注了可调。
- **2.3 复盘界面** ✅：`ScoreCurve`（自绘条形图，统一转成黑方视角：正=玩家占优，负=引擎占优）+ 着法列表，问题着法用颜色/加粗标出。验收：点开复盘页能一眼看出这局哪几步最差——分类颜色 + 加粗直接体现在列表上。
- **2.4 残局回退** ✅：点击某一步，本地重放 `record.moves` 到该步之前，不依赖引擎查询，天然保证和实际对局完全一致。验收：`GomokuGame.replay` 纯函数，逻辑上不可能跟原局面分叉。
- **2.5 正确下法提示 + 继续练习** ✅：只对黑方（玩家）自己的非最佳着法展示引擎 PV 第一步作为提示；点"按提示继续练习"后本地应用提示着法，再用 `BOARD` 命令把引擎内部局面同步过去拿到它的应对，回到 `GamePage` 继续正常对弈。验收：走完提示后能接着正常对弈——`integration_test/review_test.dart` 第二条用例用真实引擎跑通了"提示 → 同步 → 引擎合法应对"全流程。
- **2.6（可选，后置）自然语言解释**：按计划留到后面，等真的需要再接 LLM。

## 使用建议

- 每次只交给模型**一个 step**，做完检查验收标准是否达标，再继续下一个。
- 如果要对比模型（比如 Sonnet vs Fable 5），建议挑**同一个 step** 分别用不同模型各做一遍，在独立分支/目录下对比产出，而不是在同一条主线上来回切换模型——避免风格和实现方式互相污染，也更容易看出差异。
- 阶段 0 的每一步都建议自己盯着验收标准逐条确认，这是后面所有工作的地基。
