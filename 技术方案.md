# 象棋/五子棋练习 App —— 技术方案

> 基于 [总纲.md](./总纲.md) 中的想法整理，记录可行性讨论结论，作为后续开发的参考依据。

## 1. 项目目标

1. 象棋 / 五子棋人机对弈，AI 作为对手。
2. 对局结束后自动复盘，找出下得不好的地方，退回到该局面，由 AI 给出正确下法，用户按提示走完后可以继续从这个局面往下练习。

结论：**这两个功能在技术上完全可行**，且不需要自己发明算法——直接复用成熟的开源引擎即可。

## 2. 人机对弈：引擎选型

判断/搜索类工作**不用大语言模型**，用传统博弈搜索引擎（alpha-beta / MCTS），LLM 完全不参与胜负判断。

### 象棋（中国象棋）
- 通信协议：**UCCI**（象棋版的 UCI，标准文本协议，传局面用 FEN）。
- 推荐引擎：
  - **Pikafish（皮卡鱼）**：目前最强、维护最活跃，GPL 协议。
  - **Fairy-Stockfish**：Stockfish 的多变体分支，原生支持中国象棋规则，同样开源活跃。
  - ELEEYE / Cyclone：较老but仍可用。
- 难度分级：限制搜索深度/时间，或在最优解基础上按概率选次优着法。

### 五子棋
- 规则简单，搜索空间远小于象棋，纯 alpha-beta + 棋型模式识别即可做到很强，不需要神经网络。
- 推荐引擎：Rapfi、Embryo，或 Gomocup 比赛的开源引擎，均可直接复用。
- 也可以自己写一个中等强度引擎，作为一个独立的中型子项目（非必须）。

### 开源协议
- 上述引擎大多是 **GPL**。用户已确认可以接受，并计划本项目本身也走开源路线——这条路上不存在合规障碍，代码结构可以直接按"引擎作为子模块/独立进程"来设计，不需要绕开 GPL。

## 3. 复盘 AI 设计

参考 **lichess** 公开的 "Game Review" 实现思路（lila 仓库开源，可直接查阅设计细节）：

1. 对局中**每一步之前和之后的局面**都用引擎跑一次评估（固定深度或固定时间）。
2. 得到一条评分曲线（centipawn / 胜率）。
3. 计算每步走完后，实际着法评分相对引擎最佳着法评分的**损失值（centipawn loss）**。
4. 按损失值分档：漏着（Blunder）/ 错误（Mistake）/ 不精确（Inaccuracy）/ 好棋 / 最佳，阈值可参考 lichess 现有设计。
5. 找到损失最大的几步，作为"复盘退回残局"的锚点：退到该步之前的局面，把引擎的最佳着法/主变（PV）作为提示。

**LLM 的合理定位**：只做自然语言解释层，不参与正确性判断。引擎给出结构化结果（PV、评分变化、局面特征）后，LLM 负责把它转成人话（比如"这步丢了车的保护"）。这样即使 LLM 出错，也只是解释不到位，不会误导用户走错棋。

## 4. 跨平台开发与部署（仅自用测试，不上架）

用户目前只需要**能在自己的 iPhone 上跑起来的测试版**，不需要上架 App Store 或给外人用。

### 技术栈选择（已定：Flutter）
- 2026-07-19 用户最终拍板：**主线用 Flutter**，一套 Dart 代码出 iOS + Android；Swift/Xcode 只作为 iOS 构建与真机测试的底层通道（Flutter 打 iOS 包本来就要经过 Xcode，不算额外成本）。
- 引擎（C/C++）通过 `dart:ffi` 编译成静态库接入，已有开源棋类 App 走通这条路。
- 曾短暂考虑过纯 SwiftUI 方案（工具链更轻、调试更直接），因用户希望保留安卓可能性而放弃。

### 个人测试部署方案（不需要上架）
两种方式都不涉及 App Store 审核：

| 方式 | 成本 | 说明 |
|---|---|---|
| 免费 Apple ID + Xcode 直接安装到手机（Personal Team） | 免费 | 用 USB 或同一 WiFi，Xcode "Run" 到自己的 iPhone 即可。限制：**签名 7 天过期**，过期后需要用 Xcode 重新装一次（几分钟的事）。适合个人自用测试。 |
| Apple Developer Program（$99/年）+ TestFlight 内部测试 | $99/年 | 构建有效期 90 天，体验更顺滑，也方便以后想给朋友测试时直接加邮箱邀请，无需公开上架。 |

**建议**：先用免费 Apple ID + Xcode 直连安装的方式跑起来，成本为零、够用；如果后续觉得每周重装麻烦，或想找朋友一起测，再升级到 $99 的开发者账号走 TestFlight。

## 5. 项目复杂度评估

中等偏上，但**大部分是集成工作，不是算法发明**：
- 直接复用现成开源引擎（Pikafish / Fairy-Stockfish / Rapfi），不需要自己写强 AI。
- 主要工作量在：FFI 桥接调试、棋盘 UI 交互、对局记录 / 复盘状态管理、复盘界面（着法树 + 评分曲线展示）。
- 如果想自己写五子棋引擎，是个不错的中型子项目，但非必须，可以后期再做。

## 6. 模型分工建议

| 任务 | 建议模型 | 理由 |
|---|---|---|
| 架构设计（引擎桥接方案、复盘数据结构）| Opus | 需要一次性权衡清楚，减少返工 |
| 日常功能开发（Flutter UI、FFI 调用、复盘界面）| Sonnet（当前）| 性价比最高，承担大部分编码工作 |
| 疑难调试（FFI 崩溃、引擎协议对不上）| Opus | 复杂度高、试错成本大时再升级 |
| 琐碎任务（样式微调、文案、简单重构）| Haiku | 省额度 |

- **Fable 5**：目前对其相对 Sonnet/Opus 的具体强项没有足够可靠信息做定位，不建议押它做主力，可等额度富余时小范围试用对比。
- **Kimi 3**：非 Claude 模型（月之暗面出品），Claude Code 原生模型列表中没有，除非额外配置 API 代理，否则当前环境用不了，予以放弃。
